### Summary

CanopySense is a satellite-based vegetation monitoring system for oil palm plantations. Instead of sending people out to manually inspect blocks, contractors use CanopySense to automatically pull satellite imagery from Google Earth Engine, process it, and store the results — NDVI, EVI, and four other vegetation indices — directly in their own database.

The v1.0 release covers the full pipeline: from the moment a contractor's server sends a request, all the way to processed data landing in their PostGIS database. It supports two ways to trigger processing — a scheduled run that handles all plantation blocks at once, and an on-demand trigger when a user uploads a new area. Once the system is set up on the contractor's server, it runs on its own. No ongoing involvement from the CanopySense team is needed for normal operations.

# 2. Original Intent

### Problem Statement

Plantation managers need regular vegetation health data to make decisions about replanting, fertilizer timing, and block prioritization. Getting that data traditionally meant hiring field surveyors or buying expensive commercial satellite subscriptions. Neither is practical at scale.

The core problem: satellite imagery is publicly available through programs like Sentinel-2 and Landsat, but turning that raw imagery into useful per-block statistics — cloud masking, band harmonization, quality gating — requires infrastructure that most plantation operators simply don't have in-house.

### Target Users

Two distinct groups:

**Contractors (operational users)** — plantation companies with their own land management systems and databases. They run CanopySense on their own servers. They never see the satellite processing details — they just see results in their database.

**Administrator (system operator)** — the CanopySense team. Manages API keys, monitors the Cloud Function, and updates the processing engine when needed. Contractors don't need to contact the admin for normal operations.

### Success Metrics

The goals set at the start of this project:

- Automated weekly satellite data with no manual steps from the contractor
- Support for multiple sensors — Sentinel-2 as primary, Landsat 8/9 as fallback
- Processed vegetation indices stored in the contractor's own database (they own the data)
- System resilient enough to handle partial failures without losing the full run
- Contractor script deployable once, never needing updates as the cloud side evolves

### Strategic Goal Alignment

The project supports a commercial B2B model: CanopySense provides the processing infrastructure, contractors pay for access. The architecture was deliberately designed to minimize ongoing support burden. Once a contractor is onboarded, the admin's job is limited to key management and monitoring — not hand-holding.

# 3. Solution Narrative

### High-Level Solution

The system is split into two parts that communicate over HTTPS.

On the contractor's side: a single Python script (`patcher_local.py`) reads plantation block geometries from their local database, packages them into a request, sends it to the Cloud Function, and writes the results back. This script is stable — deployed once and doesn't change as the cloud side evolves.

On the admin's side: a Google Cloud Function (`patcher_cloud`) receives the request, validates the API key against a registry stored in Secret Manager, calls Google Earth Engine to process the satellite imagery, and returns vegetation index records. All the processing intelligence lives here. When the engine improves or new features are added, only this side changes — the contractor's script doesn't need to be touched.

### Key Design Decisions

| Decision                             | What Was Decided                                                                                                          | Why                                                                                                                                                                                                                                                                                                               |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Block data source (Option B)**     | Patcher-Local reads blocks from the contractor's local PostGIS and sends them as GeoJSON in the request body              | Earlier design had the Cloud Function connect directly to the contractor's database. That required either exposing their DB to the internet or maintaining a bore tunnel — both unacceptable for production. Option B removes the dependency entirely: the Cloud Function has zero outbound database connections. |
| **Thin client / thick server**       | All processing logic lives in the Cloud Function; Patcher-Local is a stable, minimal client                               | The admin can update the engine, change quality thresholds, or add new sensors without the contractor ever touching their script.                                                                                                                                                                                 |
| **Afdeling-level batching**          | Scheduled runs group blocks by afdeling, one Cloud Function call per batch                                                | Limits payload size per request and isolates failures — if one afdeling's batch fails, the rest of the run continues normally.                                                                                                                                                                                    |
| **Presence check over insert count** | Batch success is determined by querying whether blocks are actually in `satellite_data`, not by counting inserted rows    | A re-run on already-processed data inserts 0 rows but the data is still there. Counting inserts would incorrectly mark those batches as failed.                                                                                                                                                                   |
| **patcher_run_log as retry memory**  | Every batch writes its status to a local log table; the next scheduled run checks this before deciding what to re-process | Without this, a batch that failed on Wednesday would cause Thursday's run to re-process everything — including what already succeeded.                                                                                                                                                                            |

### User Flow Realization

Two main flows in production:

**Scheduled run (weekly/daily via cron):**
The contractor's cron job calls `python3 patcher_local.py`. The script loads all blocks from their PostGIS, groups them by afdeling, and sends each group to the Cloud Function in turn. Each batch result is logged. At the end, any failed batches are marked for retry on the next run. The contractor doesn't interact with any of this — they just see results in their database afterward.

**Upload trigger (when a user adds a new area):**
The contractor's application calls `python3 patcher_local.py --block-id 42`. Only that one block is processed immediately. Fast, lightweight, no impact on the scheduled run.

### Constraints & Considerations

The Cloud Function is deployed with a single-instance limit (`--max-instances=1`). This is intentional — the current engine uses process-global state for output directory handling and timeout management, which isn't safe across concurrent instances. In practice, scheduled runs are weekly and upload triggers are single-block, so one instance handles the load comfortably. This constraint is documented as technical debt for a future refactor.

# 6. Testing & Validation

### Testing Summary

Testing happened in three layers:

**Layer 1 — Local simulation (Level 1):** Full pipeline running on a single laptop using `functions-framework`. Verified end-to-end data flow from Patcher-Local through a local Cloud Function instance to Docker PostGIS. Passed.

**Layer 2 — Hybrid simulation (Level 2):** Patcher-Local on laptop → real Google Cloud Function → real Google Earth Engine → local Docker PostGIS via bore tunnel. Verified that GEE authentication, Secret Manager key resolution, and PostGIS ingestion all work together on real infrastructure. Passed.

**Layer 3 — Contract and functional tests (ANT-STR-003-v0.9):** API contract verification against the live deployed Cloud Function, engine_launcher backward compatibility, and Cloud Function environment variable audit. Phases A (API contract), I (backward compat), and J (no DB in cloud) passed. Phases B–H and K–P remain pending contractor PostGIS availability.

**Layer 4 — Docker integration tests (ANT-STR-003-v0.10):** Full isolated test environment using Docker Compose + seeded PostGIS + mock_cloud.py. Covered Phase 0 (pre-flight), Phase A (Docker infra and init.sql), Phase B (mock cloud response contract), Phase C (full local patcher-through-mock pipeline), Phase D (live cloud integration with real GEE), Phase E (PostGIS data verification), and Phase F (regression against map_previewer). All phases passed.

### QA Findings

Three issues were found and fixed during implementation and testing:

**ISSUE-1 (Critical — fixed pre-deployment):** The initial implementation used a two-step `PENDING → IN_PROGRESS` write for `patcher_run_log`. The `PENDING` status violated the DDL CHECK constraint — every batch would have raised a database error before making any HTTP call. Fixed by combining the write into a single `IN_PROGRESS` insert with `started_at` set atomically.

**ISSUE-2 (Minor — fixed pre-deployment):** Upload trigger mode was logging `block_id=NULL` in `patcher_run_log` instead of the actual block ID. Fixed by threading the `block_id` parameter through `_run_batch`.

**ISSUE-3 (Critical — found during Phase A-1 test execution):** The Cloud Function engine's Step 7 (PostGIS ingestion) was not guarded by the same conditional logic as the block-loading step. Since the Cloud Function has no PostGIS credentials, this caused a `500 Internal Server Error` on every valid request. Fixed by wrapping Step 7 in `if conn is not None`. Patcher-Local handles PostGIS ingestion on the contractor's side — the Cloud Function only needs to return the records.

# 7. Operational Guide

### Usage Overview

From the contractor's point of view, the system involves two things: a `.env` file with their credentials, and a Python script they run on a schedule. That's it. No logins, no dashboards, no manual satellite downloads. Results appear in their database automatically.

From the admin's point of view: onboard contractors (issue API keys, verify their database setup), occasionally check Cloud Logging, and update the Cloud Function when the engine improves. Contractors don't need to know when the cloud side changes.

### Key Scenarios

**Scenario 1 — Scheduled weekly run:**
The contractor's cron job calls `python3 patcher_local.py` at a set time. The script processes all plantation blocks grouped by afdeling, retrying any batches that failed in the previous run before starting on new ones. Results land in `canopysense.satellite_data` within a few minutes (plus GEE processing time — typically 1–5 minutes per batch).

**Scenario 2 — New area uploaded by a user:**
When a contractor's application user uploads a new shapefile for a new block, the application calls `python3 patcher_local.py --block-id 42`. Only that block is processed. The result is available in `satellite_data` within a few minutes.

**Scenario 3 — A batch fails mid-run:**
If the Cloud Function returns a 500 error or a network timeout for one afdeling's batch, the script logs it, records `FULL_FAILURE` in `patcher_run_log`, and continues with the next batch. The remaining afdelings are not affected. On the next scheduled run, the failed batch is automatically retried first.

**Scenario 4 — API key needs to be revoked:**
Admin updates the contractor's status in Secret Manager from `ACTIVE` to `REVOKED`. The change takes effect on the very next request — no redeployment, no restart. The Cloud Function returns `403 Forbidden` for any subsequent requests from that contractor.

### Edge Cases

- **Same data requested twice:** The system is idempotent by design. Re-running never creates duplicate records — if the data is already there, `satellite_data` silently skips the insert. The batch is still marked `FULL_SUCCESS`.
- **Block list changes between runs:** The system detects this via batch fingerprint comparison and treats it as a fresh batch rather than retrying the old failure. The admin gets a log warning.
- **Concurrent scheduled runs:** If two scheduled runs start simultaneously (e.g., a delayed cron overlapping with the next trigger), the IN_PROGRESS guard detects the conflict and the second run skips any afdeling already being processed.

### Fallback / Manual Process

If `patcher_local.py` fails to run:

1. Check `canopysense.patcher_run_log` for recent `FULL_FAILURE` rows — the `error_detail` column describes the cause.
2. Check Cloud Logging under `service_name=patcher-cloud` for what the Cloud Function received and processed.
3. For API key errors (401/403), contact the administrator with the error message from the log.
4. If the Cloud Function is temporarily unreachable, batches are automatically marked `FULL_FAILURE` after 3 retries and will be picked up on the next scheduled run — no manual intervention needed.

# 9. Future Considerations

### Pending Items

### Improvement Opportunities

- **Per-block error diagnostics:** Blocks that fail the quality gate currently get a generic error message. The engine logs the actual cloud cover percentage internally but doesn't surface it in the API response. Connecting those two would make it much easier to distinguish "cloudy day" failures from "no valid imagery ever" failures.
- **Estate-level batching:** The `BATCH_MODE=estate` option is in the code but falls back to single-batch behavior. Needed when contractors have multi-estate setups where afdeling IDs are not unique across estates.
- **Contractor-facing dashboard:** Right now contractors verify results by querying PostGIS directly. A simple read-only view of `satellite_data` would lower the technical bar for contractors who don't have SQL access configured.

### Technical Debt

- **Cloud Function single-instance constraint:** The engine uses process-global state for output directory patching and timeout handling. Safe at `--max-instances=1`, but needs a thread-safe refactor before horizontal scaling becomes possible.
- **`patcher_run_log` cleanup:** No automated pruning. Old rows accumulate, especially when block fingerprints change frequently. A scheduled cleanup job on the contractor's server should be added.
- **`deploy/main.py` manual sync:** The Cloud Function source (`patcher_cloud_function.py`) and the deployed copy (`deploy/main.py`) are kept in sync manually. A simple build step before deployment would eliminate the risk of them drifting apart.

### Next Version Hypothesis

v1.1 would likely focus on: thread-safe Cloud Function refactor (removes the `--max-instances=1` constraint), specific per-block error reasons from the engine, and the estate-level batching code path. None of these require any changes to the contractor's script — Patcher-Local v0.9 is designed to work with any future Cloud Function update that keeps the v1.0 API contract intact.
