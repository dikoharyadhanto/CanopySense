### Ringkasan

CanopySense adalah sistem pemantauan vegetasi berbasis satelit untuk perkebunan kelapa sawit. Alih-alih mengirim petugas untuk melakukan inspeksi manual ke setiap block, Manager perkebunan dapat menggunakan aplikasi CanopySense untuk secara otomatis mengambil citra satelit dari Google Earth Engine, memprosesnya, dan menyimpan hasilnya — Indeks vegetasi dan Canopy Cover — yang tersimpan langsung ke dalam database.

Pengembangan tahap 1 berfokus pada pengembangan integrasi sistem antara script utama engine dengan database postgis aplikasi CanopySense, ekstraksi citra satelit, sistem automatisasi proses, dan ekstraksi nilai indeks vegetasi. Script yang rilis sebagai versi 1.0 mencakup pipeline lengkap, mulai dari saat server Dasmap mengirimkan request, hingga data yang sudah diproses tersimpan di database PostGIS mereka. Sistem ini mendukung dua cara untuk memicu pemrosesan — scheduled run yang menangani semua block perkebunan sekaligus, dan trigger on-demand ketika user mengunggah area baru. Begitu sistem sudah diatur di server Dasmap, ia berjalan sendiri. Tidak perlu keterlibatan berkelanjutan dari tim CanopySense untuk operasional normal.

# 2. Implementasi Awal

### Pernyataan Masalah

Manajer perkebunan membutuhkan data kesehatan vegetasi secara berkala untuk mengambil keputusan tentang penanaman ulang, waktu pemupukan, dan prioritas block. Mendapatkan data itu secara konvensional berarti memperkerjakan surveyor lapangan dengan melakukan inspeksi rutin atau membeli langganan satelit komersial yang mahal. Keduanya tidak praktis dalam skala besar.

Masalah intinya: citra satelit tersedia secara publik melalui Sentinel-2 dan Landsat, namun mengubah citra mentah itu menjadi statistik per block yang berguna — cloud masking, harmonisasi band, quality gating — membutuhkan infrastruktur yang kebanyakan operator perkebunan tidak punya secara internal.

### Target Pengguna

Target pengguna terdiri atas dua kelompok yang berbeda:

**Pengguna Operasional** — perusahaan perkebunan dengan sistem manajemen lahan dan database mereka sendiri. Mereka menjalankan CanopySense di server mereka sendiri. Mereka tidak pernah melihat detail pemrosesan satelit — mereka hanya melihat hasil di database mereka.

**Administrator (operator sistem)** — pihak pengelola CanopySense, mencakup tim riset dari Universitas Indonesia dan pihak kolaborator, Dasmap. Mengelola API key, memantau Cloud Function, dan memperbarui processing engine saat diperlukan.

### Metrik Keberhasilan

Tujuan yang ditetapkan di tahap 1 (tahap awal) proyek ini:

- Data satelit mingguan otomatis tanpa langkah manual dari kontraktor
- Dukungan untuk berbagai sensor — Sentinel-2 sebagai primer, Landsat 8/9 sebagai fallback
- Indeks vegetasi yang sudah diproses tersimpan di database Dasmap sendiri
- Sistem yang cukup tangguh untuk menangani kegagalan parsial tanpa menghentikan seluruh run
- Script kontraktor yang di-deploy sekali, tidak perlu update saat sisi cloud berkembang

### Keselarasan Tujuan Strategis

Proyek ini mendukung model bisnis B2B komersial: CanopySense menyediakan infrastruktur pemrosesan, Dasmap menyediakan akses database dan sistem antarmuka aplikasi. Arsitekturnya sengaja dirancang untuk meminimalkan beban dukungan berkelanjutan. Begitu Dasmap sudah di-onboard, tugas admin terbatas pada pengelolaan key dan pemantauan — bukan pemantauan terus-menerus. Dengan kata lain, CanopySense dirancang dengan sistem otomatisasi tanpa interfensi pengelola yang intensif.

# 3. Narasi Solusi

### Workflow

Sistem ini terbagi menjadi dua bagian yang berkomunikasi melalui HTTPS.

Di sisi Kolaborator (Dasmap) : satu script Python (`patcher_local.py`) membaca geometri block perkebunan dari database lokal mereka, mengemas semuanya dalam sebuah request, mengirimkannya ke Cloud Function, dan menulis hasilnya kembali. Script ini stabil — di-deploy sekali dan tidak berubah seiring sisi cloud berkembang.

Di sisi peneliti dan administrator: sebuah Google Cloud Function (`patcher_cloud`) menerima request, memvalidasi API key terhadap registry yang disimpan di Secret Manager, memanggil Google Earth Engine untuk memproses citra satelit, dan mengembalikan record indeks vegetasi. Seluruh kecerdasan pemrosesan ada di sini. Ketika engine ditingkatkan atau fitur baru ditambahkan, hanya script-script yang disimpan dalam Google Cloud yang berubah — script yang diberikan ke pihak kolaborator dan disimpan dalam database dibuat se-robust mungkin dan tidak diperlukan update script yang intensif.

### Keputusan Desain Utama

| Keputusan                                       | Yang Diputuskan                                                                                                                                               | Alasannya                                                                                                                                                                                                                                                                                                                                                                     |
| ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Pengiriman data block dari Database - Cloud** | Patcher-Local membaca block dari PostGIS lokal kontraktor dan mengirimkannya sebagai GeoJSON dalam body request                                               | Desain awal mengharuskan Cloud Function terhubung langsung ke database Dasmap. Itu memerlukan eksposur DB ke internet atau membuat sebuah bore tunnel — keduanya tidak dapat dilakukan karena alasan resiko keamanan dalam produksi. Opsi alternatif yaitu dengan menghilangkan ketergantungan sepenuhnya: Cloud Function tidak memiliki koneksi database keluar sama sekali. |
| **Thin client / thick server**                  | Semua logika pemrosesan berat diserahkan pada sisi Cloud Function; Patcher-Local adalah client yang stabil dan dengan pemrosesan backend yang ringan          | Admin (UI) bisa memperbarui engine, mengubah threshold kualitas, atau menambah sensor baru tanpa Dasmap perlu menyentuh script utama secara langsung.                                                                                                                                                                                                                         |
| **Batching tingkat afdeling**                   | Scheduled run mengelompokkan block per afdeling, satu panggilan Cloud Function per batch                                                                      | Membatasi ukuran payload per request dan mengisolasi kegagalan — kalau batch satu afdeling gagal, sisa run tetap berjalan normal.                                                                                                                                                                                                                                             |
| **Presence check vs. jumlah insert**            | Keberhasilan batch ditentukan dengan melakukan query apakah block benar-benar ada di `satellite_data`, bukan dengan menghitung baris yang dimasukkan (insert) | Re-run pada data yang sudah diproses memasukkan 0 baris tapi datanya masih ada. Menghitung insert akan salah menandai batch tersebut sebagai gagal.                                                                                                                                                                                                                           |
| **patcher_run_log sebagai memori retry**        | Setiap batch menulis statusnya ke tabel log lokal; scheduled run berikutnya memeriksa ini sebelum memutuskan apa yang perlu diproses ulang                    | Tanpa ini, batch yang gagal hari Rabu akan menyebabkan run hari Kamis memproses ulang semuanya — termasuk yang sudah berhasil.                                                                                                                                                                                                                                                |

### Realisasi User Flow

Terdapat dua alur utama dalam produksi:

**1. Scheduled run (mingguan/harian via cron):**
Cron job Dasmap memanggil `python3 patcher_local.py`. Script memuat semua block dari PostGIS mereka, mengelompokkannya per afdeling, dan mengirim setiap grup ke Cloud Function secara berurutan. Hasil setiap batch dicatat. Di akhirnya, batch yang gagal ditandai untuk retry pada run berikutnya. Dasmap tidak perlu berinteraksi dengan semua ini — mereka hanya melihat hasil di database mereka sesudahnya.

**2. Upload trigger (ketika user menambah area baru):**
Server Dasmap memanggil `python3 patcher_local.py --block-id 42`. Hanya satu block itu yang diproses segera. Cepat, ringan, tidak berdampak pada scheduled run.

### Limitasi & Pertimbangan

Cloud Function di-deploy dengan batas single-instance (`--max-instances=1`). Engine saat ini menggunakan process-global state untuk penanganan output directory dan timeout, yang tidak aman digunakan lintas instance bersamaan. Dalam praktiknya, scheduled run bersifat mingguan dan upload trigger hanya satu block, sehingga satu instance menangani beban dengan baik. Batasan ini didokumentasikan sebagai technical debt untuk refactor di masa depan.

# 6. Pengujian & Validasi

### Ringkasan Pengujian

Pengujian dilakukan dalam tiga lapisan:

**Lapisan 1 — Simulasi lokal (Level 1):** Pipeline lengkap berjalan di satu perangkat komputer menggunakan `functions-framework`. Memverifikasi aliran data end-to-end dari Patcher-Local melalui instance Cloud Function lokal ke Docker PostGIS. Status: Berhasil

**Lapisan 2 — Simulasi hybrid (Level 2):** Patcher-Local di laptop → Google Cloud Function nyata → Google Earth Engine nyata → Docker PostGIS lokal via bore tunnel. Memverifikasi bahwa autentikasi GEE, resolusi key Secret Manager, dan ingestion PostGIS semuanya bekerja bersama di infrastruktur. Status: Berhasil.

**Lapisan 3 — Uji coba fungsional:** Verifikasi kontrak API terhadap Cloud Function yang sudah di-deploy live, backward compatibility engine_launcher, dan audit environment variable Cloud Function. Fase A (kontrak API), I (backward compat), dan J (tidak ada DB di cloud) lulus. Fase B–H dan K–P masih pending menunggu ketersediaan PostGIS Dasmap.

**Lapisan 4 — Docker integration test:** Lingkungan test terisolasi penuh menggunakan Docker Compose + PostGIS yang sudah di-seed + mock_cloud.py. Mencakup Fase 0 (pre-flight), Fase A (infrastruktur Docker dan init.sql), Fase B (kontrak response mock cloud), Fase C (pipeline patcher-through-mock lokal penuh), Fase D (integrasi cloud live dengan GEE nyata), Fase E (verifikasi data PostGIS), dan Fase F (regresi terhadap map_previewer). Semua fase berhasil.

### Temuan QA

Tiga masalah ditemukan dan diperbaiki selama implementasi dan pengujian:

**MASALAH-1 (Kritis — diperbaiki sebelum deployment):** Implementasi awal menggunakan penulisan dua langkah `PENDING → IN_PROGRESS` untuk `patcher_run_log`. Status `PENDING` melanggar CHECK constraint DDL — setiap batch akan menimbulkan database error sebelum membuat panggilan HTTP apa pun. Diperbaiki dengan menggabungkan penulisan menjadi satu insert `IN_PROGRESS` dengan `started_at`

**MASALAH-2 (Minor — diperbaiki sebelum deployment):** Mode upload trigger mencatat `block_id=NULL` di `patcher_run_log` alih-alih block ID yang sebenarnya. Diperbaiki dengan meneruskan parameter `block_id` melalui `_run_batch`.

**MASALAH-3 (Kritis — ditemukan selama eksekusi test Fase A-1):** Langkah 7 engine Cloud Function (ingestion PostGIS) tidak dilindungi oleh logika kondisional yang sama dengan langkah pemuatan block. Karena Cloud Function tidak memiliki kredensial PostGIS, ini menyebabkan `500 Internal Server Error` pada setiap request yang valid. Diperbaiki dengan membungkus Langkah 7 dalam `if conn is not None`. Patcher-Local menangani ingestion PostGIS Dasmap — Cloud Function hanya perlu mengembalikan record.

# 7. Panduan Operasional

### Gambaran Umum Penggunaan

Dari sisi Dasmap, sistem ini melibatkan dua hal: file `.env` dengan kredensial mereka, dan script Python yang mereka jalankan terjadwal. Hanya itu. Tidak perlu login, tidak perlu dashboard, tidak perlu unduhan satelit manual. Hasil muncul di database mereka secara otomatis.

Dari sisi admin (UI): onboard kontraktor (terbitkan API key, verifikasi setup database mereka), sesekali periksa Cloud Logging, dan perbarui Cloud Function saat engine ditingkatkan. Dasmap tidak perlu mengetahui informasi terkait perubahan dan pengembangan scripts di sisi cloud.

### Skenario Utama

**Skenario 1 — Scheduled run mingguan:**
Cron job Dasmap memanggil `python3 patcher_local.py` pada waktu yang ditetapkan. Script memproses semua block perkebunan dikelompokkan per afdeling, mencoba ulang batch yang gagal pada run sebelumnya sebelum memulai yang baru. Hasil tersimpan di `canopysense.satellite_data` dalam beberapa menit (ditambah waktu pemrosesan GEE — biasanya 1–5 menit per batch).

**Skenario 2 — Area baru diunggah oleh user:**
Ketika user aplikasi mengunggah shapefile baru untuk block baru, aplikasi memanggil `python3 patcher_local.py --block-id 42`. Hanya block itu yang diproses. Hasilnya tersedia di `satellite_data` dalam beberapa menit.

**Skenario 3 — Sebuah batch gagal di tengah run:**
Jika Cloud Function mengembalikan error 500 atau network timeout untuk batch satu afdeling, script mencatatnya, merekam `FULL_FAILURE` di `patcher_run_log`, dan melanjutkan ke batch berikutnya. Afdeling lain tidak terpengaruh. Pada scheduled run berikutnya, batch yang gagal otomatis dicoba ulang terlebih dahulu.

**Skenario 4 — API key perlu dicabut:**
Admin memperbarui status kontraktor di Secret Manager dari `ACTIVE` ke `REVOKED`. Perubahan berlaku pada request berikutnya — tidak perlu redeploy, tidak perlu restart. Cloud Function mengembalikan `403 Forbidden` untuk semua request berikutnya dari Dasmap tersebut.

### Kasus Tertentu

- **Data yang sama diminta dua kali:** Sistem dirancang idempoten. Menjalankan ulang tidak pernah membuat record duplikat — kalau datanya sudah ada, `satellite_data` diam-diam melewati insert. Batch tetap ditandai `FULL_SUCCESS`.
- **Daftar block berubah antar run:** Sistem mendeteksi ini via perbandingan fingerprint batch dan memperlakukannya sebagai batch baru alih-alih mencoba ulang kegagalan lama. Admin mendapat peringatan log.
- **Scheduled run bersamaan:** Kalau dua scheduled run dimulai bersamaan (misalnya cron yang terlambat tumpang tindih dengan trigger berikutnya), IN_PROGRESS guard mendeteksi konflik dan run kedua melewati afdeling yang sudah diproses.

### Fallback / Proses Manual

Jika `patcher_local.py` gagal berjalan:

1. Periksa `canopysense.patcher_run_log` untuk baris `FULL_FAILURE` terbaru — kolom `error_detail` mendeskripsikan penyebabnya.
2. Periksa Cloud Logging di bawah `service_name=patcher-cloud` untuk apa yang diterima dan diproses Cloud Function.
3. Untuk error API key (401/403), hubungi administrator dengan pesan error dari log.
4. Jika Cloud Function sementara tidak dapat dijangkau, batch otomatis ditandai `FULL_FAILURE` setelah 3 kali retry dan akan diambil pada scheduled run berikutnya — tidak perlu intervensi manual.

### Peluang Perbaikan

- **Diagnostik error per block:** Block yang gagal quality gate saat ini mendapatkan pesan error generik. Engine mencatat persentase cloud cover aktual secara internal tapi tidak memunculkannya dalam API response. Menghubungkan keduanya akan jauh memudahkan membedakan kegagalan "hari berawan" dari kegagalan "tidak pernah ada citra valid."
- **Batching tingkat estate:** Opsi `BATCH_MODE=estate` ada dalam kode tapi fallback ke single-batch. Diperlukan ketika Dasmap memiliki setup multi-estate di mana afdeling ID tidak unik lintas estate.
- **Dashboard:** Saat ini Dasmap memverifikasi hasil dengan query PostGIS langsung. 

### Technical Debt

- **Batasan single-instance Cloud Function:** Engine menggunakan process-global state untuk patching output directory dan penanganan timeout. Aman pada `--max-instances=1`, tapi perlu refactor thread-safe sebelum horizontal scaling menjadi mungkin.
- **Pembersihan `patcher_run_log`:** Tidak ada pruning otomatis. Baris lama terakumulasi, terutama ketika fingerprint block sering berubah. Sebuah scheduled cleanup job di server Dasmap harus ditambahkan.
- **Sinkronisasi manual `deploy/main.py`:** Source Cloud Function (`patcher_cloud_function.py`) dan salinan yang di-deploy (`deploy/main.py`) disinkronkan secara manual. Langkah build sederhana sebelum deployment akan menghilangkan risiko keduanya drift.
